# P3 Forge — Agent Communication Protocol v1.0

> Defines how agents talk to each other — the deliberation loops, message format,
> conversation table structure, and exact prompt patterns for every agent pair.
> This is what separates P3 Forge from a simple sequential pipeline.
> Agents don't just read each other's output — they challenge, clarify, and align.

---

## The core idea

Traditional agent pipelines: Agent A writes output → Agent B reads it → done.

P3 Forge deliberation: Agent A writes output → Agent B reads it AND challenges it →
Agent A responds → consensus reached → final output written.

This produces estimates and plans that are **negotiated between agents**, not just
generated by one agent in isolation. The result is dramatically more accurate scope,
more realistic effort estimates, and fewer QA failures.

---

## Infrastructure — p3f_agentconversation table

All agent-to-agent messages are stored here. This makes every deliberation:
- Auditable (manager can read the full conversation)
- Resumable (retry queue can replay from last message)
- Traceable (every decision has a paper trail)

| Column | Type | Notes |
|---|---|---|
| p3f_agentconversationid | Autonumber | PK |
| p3f_ticketid | Lookup → p3f_ticket | Which ticket this deliberation belongs to |
| p3f_fromagent | Text 100 | Sending agent name |
| p3f_toagent | Text 100 | Receiving agent name |
| p3f_messagetype | OptionSet | 1=Question, 2=Challenge, 3=Response, 4=Consensus, 5=Escalation |
| p3f_content | Multiline Text | Full message content (JSON) |
| p3f_round | Integer | Which deliberation round (1, 2, 3...) |
| p3f_resolved | Boolean | True when this exchange has reached consensus |
| p3f_createdon | DateTime | System-set |

### Message content schema (always JSON)

```json
{
  "question": "string — the specific question or challenge",
  "context": "string — why this question is being asked",
  "options": ["string"],       // if agent proposes alternatives
  "blocking": true,            // if true: receiving agent must respond before sender proceeds
  "my_current_answer": "string" // sender's current position (for challenges)
}
```

### Response content schema

```json
{
  "answer": "string — the response to the question",
  "revised_estimate": number,   // if hours changed
  "revised_complexity": "S|M|L|XL", // if complexity changed
  "reasoning": "string",
  "consensus": true,            // true if sender accepts this response
  "needs_another_round": false
}
```

## Flow: p3f-flow-agent-message-dispatcher

New utility flow. Called whenever one agent needs to send a message to another.

```
Input:
  {
    "ticketId": string,
    "fromAgent": string,
    "toAgent": string,
    "messageType": number,
    "content": object,
    "round": number,
    "blocking": boolean
  }

Steps:
1. INSERT p3f_agentconversation row with input data
2. if blocking = true:
   → Update p3f_ticket: set p3f_deliberationpending = true, p3f_waitingforagent = toAgent
   → Trigger the target agent flow with the message as input
   → Target agent reads message, responds, calls dispatcher back with response
   → p3f_deliberationpending = false
3. if blocking = false:
   → Insert message only — target agent will read it on its next natural trigger
```

## Deliberation 1 — Estimation Validator ↔ Architect + Requirements + Code App Expert

**This is the most important deliberation in the system.**
Three agents must reach consensus on effort before Gate 1 is shown to the manager.

### Who initiates: Estimation Validator (Phase C)

### Step-by-step

**Step 1: Estimation Validator collects initial positions**
```
Reads from Dataverse:
  - p3f_architectplan.p3f_estimatedhours         (Architect's estimate)
  - p3f_buildplan.p3f_codeappspecjson            (Code App Expert's implicit effort)
  - p3f_requirement.p3f_structuredjson           (confirmed scope)
  - p3f_ratecard for this complexity             (market anchor)

Computes its own independent estimate:
  → GPT-4o: given this requirement + this implementation plan, what is a realistic effort?
  → Returns: { own_estimate_hours, confidence, key_assumptions }
```

**Step 2: Compare all three estimates**
```typescript
const architect = architectPlan.estimated_hours;
const developer = extractDevHoursFromSpec(codeappSpec); // sum of file change estimates
const validator = ownEstimate.own_estimate_hours;

const maxDiscrepancy = Math.max(architect, developer, validator) /
                       Math.min(architect, developer, validator);

if (maxDiscrepancy <= 1.20) {
  // Within 20% — consensus immediately
  finalHours = (architect + developer + validator) / 3;
  writeConsensus();
} else {
  // Challenge round needed
  startDeliberation();
}
```

**Step 3: Deliberation round (if needed)**

Message to Architect Agent:
```json
{
  "question": "Your estimate is {architect}h for this scope. I calculate {validator}h and the developer plan implies {developer}h. The gap is {gap}%. Which specific components are you assuming will take less time than the developer has planned?",
  "context": "Requirement scope: {requirement_summary}. Developer plan covers: {files_changed} files, {queries} DV queries, {flows} flow changes.",
  "my_current_answer": "{validator}h based on historical similar tickets",
  "blocking": true,
  "round": 1
}
```

Message to Code App Expert Agent:
```json
{
  "question": "Your implementation plan implies {developer}h of work. The requirement acceptance criteria include [{unaddressed_criteria}]. Does your plan address these? If so, are they in the {files_changed} files you listed?",
  "context": "This could explain the estimate gap — if some criteria are missing from your plan, both the hours and the scope need revision.",
  "blocking": true,
  "round": 1
}
```

Message to Requirements Agent:
```json
{
  "question": "The confirmed requirement states: {requirement}. The architect estimates {architect}h and the developer {developer}h — a {gap}% gap. Is there scope in the requirement that may have been interpreted differently by each agent? Specifically: {ambiguous_criteria}",
  "blocking": false,
  "round": 1
}
```

**Step 4: Agents respond**

Architect Agent response (called by dispatcher):
```
GPT-4o receives: original task + validator's challenge message
Produces response JSON explaining revised estimate or defending original
Updates p3f_agentconversation with response
```

Code App Expert response:
```
GPT-4o receives: original implementation plan + validator's question
Either: confirms coverage → no hours change
Or: identifies gap → revises plan → revised hours
Updates p3f_agentconversation with response
```

**Step 5: Consensus or escalation**
```
Estimation Validator reads both responses.
If new estimates within 20%:
  → finalHours = weighted average (architect 30%, developer 40%, validator 30%)
  → write consensus to p3f_buildplan.p3f_finalhoursestimate
  → p3f_buildplan.p3f_discrepancyflag = false

If still >20% after round 2:
  → p3f_buildplan.p3f_discrepancyflag = true
  → Gate 1 always active (manager must see the disagreement)
  → Teams card: "Agents disagree on effort: {architect}h vs {developer}h vs {validator}h.
     Full deliberation log available. [View conversation] [Set estimate manually]"
```

**Max rounds: 2. After round 2 → escalate regardless.**

## Deliberation 2 — Code App Expert ↔ Requirements Agent + Architect Agent

**Before the developer agent writes its implementation plan, it consults.**
This is what prevents "I built the wrong thing."

### Who initiates: Code App Expert Agent (at start of Phase B)

### Step-by-step

**Step 1: Code App Expert identifies ambiguities in the requirement**
```
GPT-4o call (pre-planning):
  Input: confirmed requirement + architect's component_affected list
  Prompt: "Before writing your implementation plan, identify:
    1. Any requirement statements that are ambiguous for implementation
    2. Any component boundaries unclear from the architect plan
    3. Any acceptance criteria that could be interpreted in multiple ways
  Return: { ambiguities: [{item, question, options}], boundary_questions: [string] }"
```

**Step 2: Post questions to Requirements Agent**
```json
// For each ambiguity:
{
  "question": "The requirement says 'filter should work on all plants.' Does this mean: A) Only plants the user's commodity team is responsible for, or B) All 9 plants in the worldmap?",
  "context": "This determines whether I add a client-side filter or a Dataverse query parameter.",
  "options": ["A — user's commodity plants only", "B — all 9 plants"],
  "blocking": true,
  "round": 1
}
```

**Step 3: Post questions to Architect Agent**
```json
{
  "question": "Your component map shows KanbanBoard.tsx needs to change. The useTickets hook currently fetches all tickets. Should I modify the hook to accept filter params, or create a new useFilteredTickets hook and keep the original for other screens?",
  "context": "Creating a new hook is safer (no regression risk) but adds ~2h. Modifying existing is riskier but faster.",
  "options": ["Modify existing hook", "Create new hook"],
  "blocking": true,
  "round": 1
}
```

**Step 4: Requirements and Architect respond**

Requirements Agent is re-triggered with the clarification question as additional context.
It reads the original requirement + the question + proposed options.
Selects the correct interpretation and responds.

Architect Agent is re-triggered with the boundary question.
It makes the architectural decision and responds.

**Step 5: Code App Expert writes its plan using the clarified answers**
```
All ambiguities resolved → write implementation plan with explicit references to answers:
  "Per Requirements Agent: filter applies to user's commodity plants only (option A)"
  "Per Architect Agent: create new useFilteredTickets hook"

This means the implementation plan is traceable to specific decisions.
```

## Deliberation 3 — PA Expert ↔ Dataverse Schema Agent

**PA Expert needs to reference columns in flow logic. DV Schema must validate they exist.**

### Trigger: PA Expert finishes its flow spec draft

**PA Expert posts to DV Schema Agent:**
```json
{
  "question": "My flow spec references these Dataverse columns: [{column_list}]. Which of these need to be created vs already exist? And are there any naming conflicts with the p3f_ prefix?",
  "context": "I need confirmation before finalising my flow configuration. If columns don't exist yet, I'll make the flow conditional on DV deployment order.",
  "blocking": true,
  "round": 1
}
```

**DV Schema Agent responds:**
```json
{
  "answer": "Columns [A, B] already exist. Column [C] is in my validated_changes — it will exist after DV deployment. Column [D] has a naming conflict: p3f_status already exists with a different type. Recommend renaming to p3f_filterStatus.",
  "revised_columns": [{"old": "p3f_status", "new": "p3f_filterStatus", "reason": "naming conflict"}],
  "deployment_order_note": "Your flow must run after DV import since it references p3f_filterStatus",
  "consensus": true
}
```

**PA Expert updates its flow spec** with the corrected column names and deployment dependency.

This eliminates the conflict class that the buildplan-consolidator was previously catching — it's now resolved *before* the consolidator runs.

## Deliberation 4 — Security Review Agent → Architect + PA Expert

**Security Review Agent is the only agent that can BLOCK other agents' output.**

### Trigger: Security Review Agent finds HIGH risk

**Message to Architect Agent:**
```json
{
  "question": "Your component map includes a change to the Dataverse security role configuration (p3f_client.p3f_autonomytier write access). This is a HIGH security risk — it affects which clients can self-promote to Autonomous tier. Is this change intentional? Can it be scoped more narrowly?",
  "context": "Risk category: Authorization escalation. Any change to tier-setting logic must be reviewed by a P3 manager regardless of client trust score.",
  "blocking": true,
  "my_current_answer": "Flagging as HIGH risk — Gate 1 will be forced",
  "round": 1
}
```

**Message to PA Expert (if risk is in flow logic):**
```json
{
  "question": "Your flow spec for p3f-flow-gate1-check includes a new condition that auto-approves tickets when p3f_agentconfidence > 0.95. This bypasses the existing gate logic. Is this intentional? If so, it requires manager review as it changes the autonomy tier behaviour.",
  "blocking": true,
  "round": 1
}
```

**Regardless of response:** Security Review Agent maintains HIGH risk flag.
The response only determines whether additional context is added to the Gate 1 card.

## Deliberation 5 — Test Agent ↔ Requirements Agent

**Test Agent confirms that its test cases cover every acceptance criterion.**

### Trigger: Test Agent generates initial test specs

**Message to Requirements Agent:**
```json
{
  "question": "I've generated test cases for these acceptance criteria: [covered_list]. I could not generate tests for: [uncovered_list]. Are these criteria still in scope? If so, can you clarify them so I can write testable scenarios?",
  "context": "Uncovered criteria are often too vague to test (e.g. 'the UI should be intuitive'). If they remain, I'll convert them to UAT manual steps instead of automated tests.",
  "blocking": false,
  "round": 1
}
```

**Requirements Agent responds:**
Either clarifies the vague criteria (enabling automated tests) or confirms they should be manual UAT steps.

Test Agent updates its test specs accordingly. The UAT test steps sent to the customer are more precise as a result.

## Deliberation 6 — QA Agent → Code App Expert (failure feedback loop)

**When QA fails, the QA Agent tells the Code App Expert exactly what's wrong.**

### Trigger: QA Agent finds failed criteria

**Message to Code App Expert:**
```json
{
  "question": null,
  "context": "QA FAILURE — retry attempt {retryCount}",
  "failed_criteria": [
    {
      "criterion": "Given a user on the Commodity Teams screen, when they select a plant filter, then only tickets for that plant appear",
      "what_was_tested": "Filtered the ticket list using p3f_ticket filter on p3f_plantid",
      "what_failed": "Filter applies correctly but the screen re-renders with all tickets after 2 seconds due to the useTickets hook polling interval overwriting the filtered state",
      "suggested_fix": "The polling update should preserve active filters. Add filter state to the polling refetch params.",
      "files_likely_involved": ["src/hooks/useTickets.ts", "src/screens/KanbanBoard.tsx"]
    }
  ],
  "blocking": false,
  "round": retryCount
}
```

**Code App Expert** receives this as the primary context for its retry.
Rather than re-reading the original requirement, it focuses on the specific failure.
This dramatically improves retry success rate.

## Phase B — corrected execution order with communications

```
Phase A: architect-agent (alone)
  ↓ (writes p3f_architectplan)

Phase B-1: pa-expert + codeapp-expert + dataverse-expert + test-agent
  (all 4 start simultaneously)

  codeapp-expert FIRST ACTION: post clarification questions to requirements + architect
    → blocking = true → waits for both responses
    → only THEN writes implementation plan

  pa-expert FIRST ACTION: post column validation to dataverse-expert
    → blocking = true → waits for response
    → only THEN finalises flow spec

  test-agent: post coverage check to requirements-agent
    → blocking = false → can write initial specs, updates when response arrives

  dataverse-expert: responds to pa-expert and runs its own schema validation

Phase B-2: security-review-agent
  (starts AFTER pa/codeapp/dv experts complete — reads all their outputs)
  Posts challenges if HIGH risk found → architect and pa-expert respond
  → security review is final in Phase B

Phase C: estimation-validator
  (starts after all Phase B agents complete)
  Collects: architect estimate + codeapp-expert hours + validator's own estimate
  Runs deliberation if discrepancy > 20%
  → 2-way challenge with architect and codeapp-expert
  → 1-way validation with requirements-agent
  → Consensus or escalation

Phase D: buildplan-consolidator
  (starts after estimation-validator reaches consensus or escalation)
  All cross-checks now much cleaner — most conflicts resolved in deliberation
```

## Deliberation rules

```
1. Max 2 rounds per deliberation pair (except QA→Dev which follows retrycount max)
2. Blocking messages must be responded to within the agent's timeout window
3. If response not received within timeout → retry queue → escalate at 2hr
4. All deliberation messages stored in p3f_agentconversation — immutable after creation
5. Consensus messages trigger the next phase automatically
6. Escalation messages always activate Gate 1 regardless of tier
7. The manager can read the full deliberation log from the Gate 1 review screen
8. Agent responses cannot contradict security-review-agent HIGH risk flags
9. Requirements Agent responses to clarification questions are binding —
   they update p3f_requirement.p3f_clarifications (new JSON column) as addenda
10. The original confirmed p3f_requirement remains immutable —
    clarifications are addenda, not edits
```
